# 视频治理平台主要技术难点文档

> 范围说明：本文仅基于 `docs/technical-design.md` 审查整理，不引用 PRD 或前后端单独技术方案。

## 1. Technical Design 审查结论

`technical-design.md` 的核心架构方向是成立的：它没有让 LLM 直接对整条视频做最终处置，而是采用“证据提取 -> 基础安全初筛 -> LLM 策略理解 -> 规则引擎聚合 -> 人审兜底 -> 申诉与数据回流”的链路。这个分层能把不可解释的模型判断拆成可追溯证据、结构化维度判断和可审计规则决策，适合内容治理这类高风险业务。

整体选型也比较务实：MVP 采用模块化单体，配合 FastAPI、PostgreSQL、Redis、Celery、MinIO/S3，降低早期运维复杂度；在模块边界上保留后续拆分 evidence worker、review worker、flywheel worker 的空间。

主要风险不在“技术栈是否能做”，而在跨模块一致性、模型不确定性、高危合规边界、人审并发锁、数据回流质量和可观测运维闭环。下面列出本项目最主要的技术难点。

## 2. 主要技术难点

### 2.1 可追溯 EvidencePackage 的生产与演进

**为什么难**

EvidencePackage 是全链路的数据基座，承载关键帧、ASR、OCR、目标检测、场景标签、初筛结果、LLM verdict、token 预算和访问策略。它同时被机审、规则引擎、人审工作台、申诉、审计和数据回流消费。

**主要风险**

- JSONB 提供灵活性，但 schema 演进、字段缺失、旧版本兼容、索引查询都会变复杂。
- 视频抽帧、ASR、OCR、目标检测的失败模式不同，必须显式记录 `modality_availability` 和 `truncated_modalities`，否则人审和 LLM 可能把“没有证据”误解为“没有风险”。
- 证据包包含敏感素材，普通证据和 CSAM 类证据必须有访问隔离和留痕。

**落地检查点**

- 为 EvidencePackage 建立版本化 schema 和兼容读取器。
- 所有下游消费只通过 schema 访问，不直接读取任意 JSONB 字段。
- 对关键字段做契约测试：`frames/asr_transcript/ocr_results/object_detections/scene_tags/pre_filter_results/llm_verdicts/access_policy`。
- 明确大对象存储边界：数据库存元数据和引用，对象存储放媒体帧和原始素材。

### 2.2 四阶段机审流水线的幂等、重试与死信闭环

**为什么难**

视频审核流水线是长耗时、多外部依赖、多阶段持久化流程。方案使用 Celery chain，把证据提取、初筛、LLM 审查、规则聚合拆为独立任务，并通过 dead letter 记录永久失败。

**主要风险**

- 某阶段重试时不能重复产生副作用，比如重复写 EvidencePackage、重复创建人审任务、重复触发告警。
- 已完成阶段必须可复用，否则 LLM 或抽帧失败会导致前面昂贵任务重复执行。
- Celery、Redis Streams 和数据库状态之间要避免“双写不一致”。
- 死信队列只有记录还不够，还需要可操作的重试、放弃、告警和根因定位流程。

**落地检查点**

- 每个阶段输出都绑定唯一业务 ID，如 `content_id`、`evidence_package_id`、`task_id`。
- 对 `upload/decide/appeal` 等关键写入端点强制 `Idempotency-Key`。
- 死信表必须支持人工重试、标记放弃、按异常类型聚合。
- 用集成测试覆盖“阶段 3 失败后重试不重新抽帧”的场景。

### 2.3 LLM 策略审查的可靠性、安全性与成本控制

**为什么难**

LLM 只负责“理解和归因”，不负责最终处置，这是正确边界。但它仍然是最不稳定、最贵、最容易被输入污染的环节。

**主要风险**

- Prompt 注入防护目前在方案中主要描述为 `<user_content>` 分隔，实际还需要输入净化、输出 schema 校验和异常 verdict 降级。
- 长视频、多模态证据可能超出上下文预算，需要稳定的 token 裁剪策略。
- 并行执行多个策略维度会放大 LLM 成本和外部 API 限流风险。
- LLM 不可用时全部变为 `UNCERTAIN` 会把压力转移给人审队列。

**落地检查点**

- LLM 输出只能是 `DimensionVerdict`，不得包含最终处置动作。
- 输出 schema 校验失败、超时、注入疑似、模型不可用时统一降级为 `UNCERTAIN` 并路由人审。
- 按视频时长和模态重要性定义 token 预算，记录 `token_budget_used/token_budget_limit/truncated_modalities`。
- 建立 LLM 成本、耗时、失败率、熔断状态和 `UNCERTAIN` 比例监控。

### 2.4 规则引擎聚合与三层决策枚举映射

**为什么难**

系统同时存在 L1 维度判断、L2 规则引擎决策和前端/人审看到的处置建议。技术方案已经明确：LLM 输出 `VIOLATION/NO_VIOLATION/UNCERTAIN`，规则引擎输出 `auto_pass/auto_block/needs_human_review/critical_escalate`，人审 MVP 只提交 `pass/block`。

**主要风险**

- 枚举映射一旦散落在前后端，会出现同一状态被不同模块解释为不同含义。
- `needs_human_review` 若被前端硬映射成 `block`，会引入系统性拦截偏差。
- 阈值、法域、维度配置、策略版本和规则版本必须绑定到每次决策，否则无法复盘。

**落地检查点**

- 规则引擎是最终处置责任主体，禁止 LLM 直接返回 `pass/block/remove`。
- 后端返回 `machine_recommendation`，`needs_human_review` 映射为 `uncertain`。
- 每条 `machine_review_results` 必须记录 `policy_version`、`rule_version`、`model_version` 或模型版本引用。
- 策略阈值变更必须进入版本生命周期：draft -> shadow -> active -> archived。

### 2.5 策略注册表热加载与零改造扩展

**为什么难**

方案要求新增审核维度时，不修改决策引擎、人审工作台、申诉闭环和审计核心代码。这要求策略、配置、UI、SoR、审计和回流都围绕 `dimension_id` 建模。

**主要风险**

- 运行时热加载配置可能造成并发读写竞态。
- “零改造”容易在第一批特殊策略中被打破，出现硬编码分支。
- 新策略上线需要 Shadow 验证、灰度放量、回滚和差异报告，否则配置化反而提高事故半径。

**落地检查点**

- StrategyRegistry 使用 copy-on-write 快照，运行中读请求只读稳定快照。
- 新策略必须经过 Shadow 验证，达到差异阈值后再激活。
- 前端优先使用 DynamicForm 和默认渲染器，自定义渲染器作为第二层扩展。
- 增加测试：注册表热加载不丢配置、旧请求不读到半更新配置。

### 2.6 人审任务队列的并发锁、SLA 与独立性约束

**为什么难**

人审工作台是系统兜底能力。任务领取需要满足优先级、SLA、法域权限、技能、疲劳上限、独立性排除和并发锁。

**主要风险**

- 高并发领取下不能出现同一案件被两个审核员处理。
- 锁超时、心跳续约、WebSocket 状态和数据库状态可能不一致。
- 申诉二审、质检复审必须排除原审核员，否则破坏独立性。
- CSAM 曝光上限和强制休息是合规要求，不只是 UI 提醒。

**落地检查点**

- 领取任务使用 `SELECT FOR UPDATE SKIP LOCKED`，并覆盖并发测试。
- 锁状态以数据库为权威，Redis/WebSocket 只做加速和通知。
- 心跳超时释放后，前端必须停止提交旧锁任务。
- 反疲劳校验放在分配前置逻辑，不能只依赖前端。

### 2.7 WebSocket 实时状态的一致性与重连补偿

**为什么难**

WebSocket 推送锁续约、SLA 告警、任务重分配、强制休息和高危告警。这些消息影响审核员能否继续操作当前案件。

**主要风险**

- 多 FastAPI 实例下，本地 WebSocket 连接表无法跨实例广播，必须依赖 Redis Pub/Sub。
- 消息丢失会导致前端展示过期锁状态或过期 SLA。
- MVP 重连后全量 invalidate 查询可行，但用户体验和流量成本较高；V2 需要 Redis Stream 补发。

**落地检查点**

- 消息信封必须包含 `type/payload/timestamp/correlation_id`，前端按 `correlation_id` 去重。
- 后端同时支持 `HEARTBEAT/HEARTBEAT_ACK`，兼容 `PING/PONG`。
- 断线重连后必须重新拉取任务详情和锁状态。
- 对 `task_lock_expired/task_reassigned/kill_switch_activated` 做 E2E 测试。

### 2.8 申诉、审计与最终一致性的恢复链路

**为什么难**

申诉改判会触发内容可见性恢复、账号处罚回滚、质检负反馈和样本回流。方案中定义改判事务强一致，恢复链路最终一致。

**主要风险**

- 改判成功但恢复链路失败，会造成创作者侧状态和内部裁决不一致。
- 审计日志链式哈希要求 append-only，如果补偿动作缺审计会破坏追溯完整性。
- 申诉通道内“不可加重处置”的硬约束需要在服务层校验。

**落地检查点**

- 改判决策与审计日志同事务提交。
- 恢复链路拆成可重试步骤，每步有幂等键和审计事件。
- 按 `content_id` 可追溯摄取、机审、人审、申诉、最终处置。
- 审计链完整性校验纳入运维接口和定期任务。

### 2.9 数据回流 Flywheel 的样本质量与反馈偏差

**为什么难**

数据回流是模型持续优化的基础，但回流数据天然带有机审、人审、申诉和策略版本的偏差。

**主要风险**

- 只回流被拦截或争议样本，会让训练集分布偏向难例和负例。
- 申诉改判样本质量高，但数量少且具有选择偏差。
- 策略版本变化后，旧样本标签语义可能失效。
- JSONL 导出若不携带版本和来源，后续训练不可复现。

**落地检查点**

- 每条 `flywheel_samples` 记录 `source_type`、`policy_version`、`model_version`、`rule_version`。
- 训练前必须通过质量门控，区分 ground_truth、disagreement、golden、correction。
- Shadow 报告用于观察策略/模型变化，不能只看总体准确率。
- 数据导出必须可复现：同一输入、同一策略版本、同一模型版本可还原标签来源。

### 2.10 高危内容与安全边界

**为什么难**

技术方案中一方面把“高危强制上报（CSAM / NCMEC）”列为本期暂不交付，另一方面又设计了 CSAM 独立加密桶、CSAM 曝光计数和 `P0_CSAM_REPORT_OVERDUE` 告警。这说明高危能力在设计里已经进入关键路径，但范围边界需要进一步明确。

**主要风险**

- MVP 若真的不交付强制上报，但实现了 CSAM 检测和存储，就会出现检测到高危内容却没有完整处置闭环的合规风险。
- CSAM 原始素材、证据帧、日志、审计事件不能进入普通存储和普通日志。
- 权限模型必须支持 break-glass、双人授权或最小权限访问，否则独立桶只是名义隔离。

**落地检查点**

- 明确 MVP 是否启用 CSAM 检测；一旦启用，必须启用隔离存储、访问审计、人工干预和告警闭环。
- 普通证据 API 必须根据 `access_policy` 屏蔽 CSAM 素材。
- CSAM 类操作单独审计，日志不记录可识别内容特征。
- 反疲劳指标不能与绩效负向绑定。

### 2.11 前端人审工作台的性能、韧性与操作效率

**为什么难**

人审工作台要求审核员在一个界面内完成视频播放、命中点定位、多模态证据查看、机审解释理解、处置提交、疲劳保护和快捷键操作。

**主要风险**

- 视频、证据帧、OCR bbox、ASR 时间轴和目标检测叠加会造成渲染压力。
- 任一面板崩溃不能阻断提交路径。
- 快捷键提高效率，也可能导致误操作，必须与确认、草稿和回滚策略配合。
- 创伤屏蔽既要保护审核员，又不能影响必要证据判断。

**落地检查点**

- 三层错误边界：全局、路由、面板。
- 视频懒加载、证据预加载、长列表虚拟滚动、请求去重。
- `VideoPlayer/EvidenceViewer/DispositionPanel` 做独立降级。
- Playwright 覆盖完整人审流程、锁过期、提交失败回滚、创伤屏蔽切换。

### 2.12 可观测性与运维闭环

**为什么难**

该系统不是单一 API 服务，而是异步流水线、模型调用、人审队列、申诉、回流和 WebSocket 推送的组合。问题发生时，需要快速定位是模型、队列、数据库、Redis、存储、前端还是人审容量不足。

**主要风险**

- 只监控基础设施，不监控业务漏斗，会看不到审核质量退化。
- 熔断器打开、死信积压、人审队列积压、申诉改判率上升之间存在连锁反应。
- 没有 runbook 的告警会变成噪音。

**落地检查点**

- 核心指标至少覆盖：流水线耗时、LLM 耗时与 token、决策分布、人审队列、SLA 违规、人审推翻率、申诉改判率、死信、熔断器失败率、黄金题准确率。
- 为 `P1_DEAD_LETTER_ACCUMULATION`、`P1_CIRCUIT_BREAKER_OPEN`、`P2_QUEUE_BACKLOG` 建立处理手册。
- 大屏展示三阶段漏斗转化和异常告警，不只展示总量。
- 所有异步任务带 `correlation_id/content_id/evidence_package_id`，方便串联日志。

## 3. 建议优先级

| 优先级 | 建议事项 | 原因 |
|---|---|---|
| P0 | 冻结 EvidencePackage、决策枚举、分页响应、WebSocket 消息信封等契约 | 这些契约一旦漂移，会让前后端、人审、回流和审计同时返工 |
| P0 | 跑通一条端到端垂直切片：上传 -> 证据包 -> 初筛 -> LLM stub -> 决策 -> 人审 -> 审计 -> 回流 | 先证明关键链路闭环，再替换真实模型和复杂 UI |
| P0 | 明确 CSAM / high-risk MVP 边界 | 设计中已经包含隔离桶和告警，必须避免“检测了但没闭环” |
| P1 | 建立流水线幂等、死信重试和阶段复用测试 | 这是异步系统稳定性的底座 |
| P1 | 实现 LLM 输出校验、token 预算和降级策略 | 控制模型不可用、超时、注入和成本风险 |
| P1 | 压测人审任务领取与锁续约 | 防止重复分配和过期提交 |
| P1 | 补齐 observability dashboard 与告警 runbook | 保证上线后问题可定位、可恢复 |

## 4. 推荐落地顺序

1. **契约冻结**：EvidencePackage v1、DimensionVerdict、PolicyDecision、HumanReviewDecision、PaginatedResponse、WSMessage。
2. **垂直切片**：用 mock 模型跑通完整审核闭环，验证事务、审计、状态机和 API。
3. **可靠性底座**：幂等键、Celery 阶段复用、死信队列、熔断器、降级策略。
4. **人审工作台**：任务领取、锁续约、证据展示、处置提交、黄金题同步评估。
5. **策略扩展机制**：注册表、Shadow、灰度、回滚、动态表单。
6. **数据飞轮**：样本质量门、JSONL 导出、Shadow 报告、训练集版本化。
7. **生产化**：监控告警、runbook、安全审计、CSAM 隔离验证、K8s 扩缩容。

## 5. 上线前关键验收清单

- 机审四阶段每个阶段都可独立重试，且不会重复产生副作用。
- LLM 不可用时内容不会静默通过，而是进入人审或高危人工干预。
- 并发领取 100+ 任务时不出现重复分配。
- WebSocket 断线重连后，任务详情和锁状态能恢复到权威状态。
- EvidencePackage 字段完整率和 schema 校验通过率达到上线门槛。
- 每条最终决策都能追溯到证据包、策略版本、规则版本、模型版本和审计链。
- 策略热加载、Shadow、激活、回滚均有自动化测试。
- CSAM/critical 相关存储、API、日志和前端展示均通过隔离验证。
- 人审提交、申诉提交、上传接口均支持幂等。
- 契约测试、集成测试和人审 E2E 测试纳入 CI。
